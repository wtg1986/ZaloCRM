🎯 Cốt lõi đáng nhớ

• 1 entity Flow handle cả 2 shape: Campaign (manual + list) và Trigger-flow (event/cron + filter).

• Step Executor unified chạy mọi loại step: send / wait / if / tag / notify.

• Atom embed vào step.payload.atoms[] — chưa cần Block entity riêng.

• Smart Nick Picker là gate trung tâm: rule throttle + delay + permission + dedup tất cả check ở đây.

• Counter ở level NICK, không phải campaign. Nick X dùng cho 3 campaign khác nhau, counter cộng dồn.

📐 Smart Nick Picker — gate đầy đủ

Khi walker pick nick cho 1 row, scan selectedNickIds[] và filter qua 6 rule. Nick nào pass cả 6 → vào candidate pool. Tie-break: pick nick có dailyFriendReqSent thấp nhất.

  1. liveStatus = connected — nick đang online qua zalo-pool
  2. dailyFriendReqSent < cap(tier) — chưa đầy quota hôm nay (tier: new=10, warming=20, trusted=30)
  3. NOW - lastFriendReqSentAt ≥ random(15min, 45min) — per-nick delay gate, cross-campaign
  4. activeHours 6h-22h — không gửi đêm
  5. permission via ZaloAccountAccess ≥ chat — sale tạo campaign có quyền dùng nick
  6. NOT already friend với UID này — check Friend table cả org

🔁 Cross-nick friendship recency rule

Khi KH 0908.xxx.123 đã là friend của nick Y rồi (không phải nick X đang muốn gửi), check lastInboundAt max qua tất cả Friend records của contact đó:

• NOW - lastInboundAt > X ngày (KH im lặng) → VẪN GỬI từ nick X (re-approach)

• NOW - lastInboundAt ≤ X ngày (KH đang active với nick khác) → SKIP, status=active_with_other_nick

• Chưa từng inbound → coi như im lặng, gửi bình thường

X là input config trên campaign (default 30 ngày, sale có thể override).

⏱ Per-nick delay independence

Nick 1 vừa gửi xong → wait 15-45 phút random. Trong thời gian wait, nick 2-10 vẫn AVAILABLE và được walker pick bình thường. Walker không "wait nick 1", chỉ skip nick 1 trong tick này.

Hệ quả: 10 nick song song có thể spread ra 30 send / giờ team (nếu delay average 20 phút × 10 nick = 1 send mỗi 2 phút từ team).

📊 Quality tier — nick warmup

Nick mới < 30 ngày: tier = new, cap = 10 friend-req/day

Nick 30-90 ngày: tier = warming, cap = 20 friend-req/day

Nick > 90 ngày: tier = trusted, cap = 30 friend-req/day

Admin có thể override per-nick qua friend_req_cap_override.

🧲 Sticky nick assignment

Khi nick X gửi friend-req cho SĐT 0908...123 → set RunRow.assignedNickId = X.

Nếu KH accept (3 ngày sau) → tạo Friend record với ownerNickId = X.

Sau đó S1 welcome flow chạy → phải dùng nick X gửi msg, KHÔNG reassign sang nick khác. Tag CRM gắn vào Contact (chung), nhưng thread conversation thuộc nick X.

🚨 Edge cases khi code

• Duplicate phone trong 1 list import → dedup at parse (unique(campaignId, phoneE164))

• 2 campaign cùng share 1 nick → throttle gate per-nick handle

• Server restart giữa flow → walker resume rows với nextRunAt < NOW

• Sale gửi msg tay đè lên flow đang chạy → hook detect outbound manual → auto-pause Run của contact đó

• Nick disconnect giữa campaign → row.status = queued, retry sau khi nick reconnect

• Dry-run preview: lookup 10 SĐT random trước khi run thật, hiện estimate (có Zalo / không / đã friend / sẽ skip)

• Quiet hours 22-6h: walker skip pick, ngủ tới 6:00 sáng
